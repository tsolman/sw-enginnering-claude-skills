# Product Discovery Framework

Based on Marty Cagan's "Inspired: How to Create Tech Products Customers Love"

## Core Principle

> "The purpose of product discovery is to quickly separate the good ideas from the bad. The output of product discovery is a validated product backlog."

Discovery is about **de-risking ideas before building**. Strong teams test 10-20 product ideas weekly using prototypes, not production code.

## The Four Critical Risks

Every product idea must be validated across these dimensions:

| Risk | Question | Validation Method |
|------|----------|-------------------|
| **Value** | Will customers want this? | Customer interviews, fake door tests, demand testing |
| **Usability** | Can users figure it out? | Prototype testing, usability studies |
| **Feasibility** | Can we build it? | Technical spike, engineer assessment |
| **Viability** | Does it work for business? | Stakeholder review, compliance check, financial model |

## Discovery Process Phases

### 1. Framing
Align team on the problem before exploring solutions.

**Opportunity Assessment** (for smaller features):
- What business objective does this address?
- How will we measure success (key results)?
- What customer problem are we solving?
- What type of customer are we targeting?

**Customer Letter** (for larger initiatives):
Write a hypothetical press release or customer testimonial from the future. Forces clarity on the value proposition.

**Startup Canvas** (for new products):
Cover problem, solution, key metrics, unique value proposition, unfair advantage, channels, customer segments, cost structure, revenue streams.

### 2. Ideation
Generate multiple solution alternatives before committing.

**Techniques:**
- Customer story mapping
- Design sprints
- Competitive analysis
- Technology-enabled brainstorming (what's now possible?)
- Customer co-creation sessions

**Rule:** Never fall in love with your first solution.

### 3. Prototyping
Create artifacts for testing—not shipping.

**Prototype Types:**

| Type | Fidelity | Best For | Time to Build |
|------|----------|----------|---------------|
| Paper/Wireframe | Low | Early concept validation | Hours |
| Clickable mockup | Medium | Usability testing | Days |
| Wizard of Oz | High (fake backend) | Value testing | Days |
| Concierge | High (manual backend) | Service validation | Days |
| Live data prototype | High | Feasibility + value | 1-2 weeks |

**Principle:** Prototype to learn, not to ship.

### 4. Testing
Validate against the four risks with real users/stakeholders.

**Qualitative Testing:**
- User interviews (5-10 users often sufficient)
- Usability tests (watch, don't guide)
- Customer discovery calls

**Quantitative Testing:**
- A/B tests
- Fake door tests (measure intent before building)
- Analytics on prototype interactions

## Discovery Anti-Patterns

Avoid these common failures:

1. **Building to learn** — Use prototypes instead of production code
2. **Validating with stakeholders only** — Real customers reveal real problems
3. **Asking "would you use this?"** — Watch behavior, don't trust opinions
4. **Skipping feasibility** — Engineers must be in discovery from day one
5. **One-and-done testing** — Expect 3-5 iterations minimum

## Missionaries vs. Mercenaries

**Mercenary teams** build what they're told. They:
- Take feature requests at face value
- Measure success by shipping
- Own output, not outcomes

**Missionary teams** solve problems. They:
- Challenge assumptions
- Measure success by customer/business impact
- Own outcomes, empowered to find solutions

**Goal:** Build a team of missionaries who believe in the product vision.

## Discovery Cadence

Weekly rhythm for mature teams:

| Day | Activity |
|-----|----------|
| Monday | Review learnings, prioritize tests |
| Tue-Thu | Run experiments, prototype, test with users |
| Friday | Synthesize findings, update backlog |

## Output: Validated Backlog

Discovery produces a backlog of items that:
- Have evidence of customer value
- Have been tested for usability
- Have been assessed for feasibility
- Have been cleared for business viability

This is what enters delivery—not ideas, but validated solutions.

## Key Mindset Shifts

| From | To |
|------|-----|
| Feature roadmaps | Outcome-based planning |
| "Build it and they will come" | Evidence before commitment |
| PM as feature definer | PM as problem expert |
| Engineer as implementer | Engineer as inventor |
| Design as decoration | Design as strategic |
